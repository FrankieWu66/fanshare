## Problem Statement

NBA fans who follow player performance have no way to take a continuous,
non-binary financial position on how well a player performs over time.
Current options are all binary: you win or lose a single bet. There is no
equivalent of "buying and holding" a player's upside the way you'd hold a
stock.

The DFS/sportsbook workaround: sophisticated bettors repeatedly place the
same high-probability prop bet (e.g., "LeBron James over 10 points") game
after game as a proxy for continuous exposure. This is tedious, capital-
inefficient (money cycles every 24 hours), and pays the house vig on every
single transaction. They are manually simulating a long position that a
single token purchase should replace.

FanShare is a Solana-based DEX where fans trade tokens representing NBA
players. Price is set by AMM mechanics (supply/demand). A stats reference
index — calculated from real-time player performance data — gives traders
a benchmark to evaluate whether a token is over or underpriced relative
to actual on-court performance.

## Status Quo

Sports bettors who want continuous player exposure have two bad options:

1. **Manual prop rolling:** Bet the same prop (LeBron over 10 pts) every
   game. Requires daily action, ties up capital in 24-hour cycles, pays
   1-5% vig to the house on every single bet. 30+ separate transactions to
   hold a "position" for a month.

2. **Season-long fantasy leagues:** No real money at stake for most players.
   Locked-in rosters, no intra-season liquidity. You can't sell your LeBron
   position mid-season if he gets injured.

The workaround cost: time (daily bets), money (vig), and frustration
(no single interface to track a player position over time).

## Premises

1. **Core value = continuous, non-binary exposure** to player performance.
   Not collectibles. Not binary bets. The token is a hold-able position.

2. **v1 target = crypto-native Solana users.** Embedded wallet + fiat onramp
   (for casual NBA fans) is v2 scope — not a shortcut, a deliberate expansion.

3. **Liquidity bootstrapping is the critical launch constraint.** No founder
   capital. Solution must be bonding curve (no LPs required) or investor-seeded
   pools post-fundraise.

4. **Stats oracle = reference index only.** An existing Solana-compatible sports
   data feed must be identified before mainnet launch. Oracle does NOT directly
   set price — it informs traders. This must be validated during technical design.

5. **Price is set by bonding curve supply/demand only.** Stats index is a
   benchmark shown to traders — it does not feed into the price calculation.
   Legal review recommended before any public token launch or LP incentive
   programs. A more sophisticated stats-anchored pricing model is possible
   in future versions but requires legal clearance first.

## Approaches Considered

### Approach A: Fork-First (Discarded for v1)
Fork Orca/Raydium AMM. Deploy player SPL tokens. Off-chain stats API as
reference. Ships fastest but leaves liquidity bootstrapping unsolved — requires
seeded pools which requires capital we don't have. Deferred to potential v2
once funded.

### Approach B: Custom Anchor Program (Future Architecture)
Full custom Rust/Anchor program. On-chain oracle (Switchboard). Custom AMM
mechanics. Most defensible long-term but XL effort, requires security audit
($20-50k), and is overbuilt before PMF is confirmed. Target for v2/v3 after
devnet validation proves demand.

### Approach C: Launchpad Model (Chosen — Pump-and-Play)
Each player token launches on a bonding curve. No LPs required — built-in
liquidity from day one. Price rises with buys, falls with sells. Stats
reference index shown alongside current price. Once a player token hits a
market cap graduation threshold (e.g., 69 SOL), it migrates to a Raydium
pool automatically for deeper liquidity. This is pump.fun's proven Solana
model applied to real-world performance assets.

---

## Recommended Approach

**Launchpad Model: Bonding Curve → Raydium Graduation**

**Why this approach:**
- Solves liquidity bootstrapping without any capital requirement
- pump.fun proved this model at $2B+ volume on Solana — the mechanics are
  battle-tested and familiar to the target crypto-native user
- Differentiator: pump.fun tokens are pure speculation. FanShare tokens are
  speculation anchored to real performance data (the stats index). That's
  a genuinely new product on top of a proven launchpad mechanic.
- Devnet parity: build it once, run it on devnet, flip to mainnet

**Key components:**

1. **Player token factory:** SPL token creation for each NBA player. Each
   token gets its own bonding curve state account. For devnet, use abstract
   identifiers (Player_23, Player_77) rather than player names to avoid right
   of publicity issues during testing.

2. **Bonding curve program (custom Anchor, Rust):** Linear curve for v1
   devnet (price = base_price + k * supply_sold). Linear is predictable and
   easy for beta users to understand — "the more people buy, the more expensive
   it gets." Constant product (x*y=k) is better for deep markets and is the
   target for post-graduation Raydium pools. Expected build time: 2-4 weeks
   for a competent Solana/Anchor developer. No ready-made audited open-source
   version exists — plan to write custom Anchor code. Treasury (SOL) accumulates
   in the program's escrow account as tokens are bought.

3. **Stats reference index (on-chain oracle account):** A dedicated Solana
   account (updated by an off-chain cron job) stores a single u64 value
   representing the index price in lamports. Update cadence: after each game
   completes (not real-time — post-game final stats only). The off-chain job
   pays the write transaction fees (~0.000005 SOL per update, negligible).

   **v1 formula (devnet):**
   ```
   composite_score = (pts_per_game × 0.5) + (ast_per_game × 0.3) + (reb_per_game × 0.2)
   // composite_score is a rolling 10-game average of the above weighted sum
   // e.g., LeBron: (27.4×0.5) + (8.1×0.3) + (7.3×0.2) = 13.7 + 2.43 + 1.46 = 17.59
   index_price_lamports = composite_score × base_multiplier
   // base_multiplier = 10,000,000 lamports (0.01 SOL per composite point)
   // e.g., LeBron: 17.59 × 10,000,000 = 175,900,000 lamports = 0.1759 SOL
   ```
   The multiplier (0.01 SOL per composite point) is a provisional constant
   set to keep index prices in a readable SOL range (0.05–0.30 SOL for typical
   starters). Adjust after devnet testing if prices feel too low or high.
   Formula is versioned; updating the multiplier or weights requires a
   new cron deployment but no on-chain program change.

   **Data source for devnet:** BallDontLie API (free, no license required,
   covers NBA stats). For mainnet: evaluate Sportradar or Stats Perform for
   licensed feeds; wrap in a Switchboard custom oracle job for decentralized
   validation.

4. **Graduation mechanism:** When a player token's bonding curve treasury
   reaches 69 SOL (provisional — may adjust based on devnet economics):
   a) Bonding curve program marks the token as "graduated" (no more buys/sells
      on the curve)
   b) A graduation transaction creates a new Raydium CPMM pool seeded with
      the treasury SOL + a fixed allocation of the remaining token supply
   c) Existing token holders' tokens are now tradeable on Raydium immediately
      — no migration of individual positions needed (SPL tokens are fungible)
   d) Graduation token allocation: 20% of remaining unsold token supply is
      paired with the treasury SOL in the Raydium pool. The remaining 80% of
      unsold supply is burned. This is consistent with pump.fun's mechanics
      and ensures the Raydium pool has reasonable token depth at launch.
   e) Failure handling: if Raydium pool creation fails, the bonding curve
      program marks the token as "graduation_pending" and the off-chain
      graduation cron retries every 60 seconds. Trades are paused during
      graduation_pending state (buy/sell buttons disabled in UI with
      "graduating..." message). Note: Raydium graduation via SDK may require
      a client-side transaction rather than an Anchor CPI — treat Raydium
      graduation as a spike item before committing to this flow.
   f) **Phasing decision:** v1 devnet ships WITHOUT Raydium graduation.
      Tokens stay on the bonding curve indefinitely during devnet testing.
      Graduation is added in v1.5 before mainnet. Success criteria below
      reflect this phasing.

5. **Frontend:** Two core screens — Market (player grid with price + stats
   index) and Trade (bonding curve chart, stats panel, buy/sell widget).

   **Wallet adapter:** `@solana/wallet-adapter-react` + `@solana/wallet-adapter-wallets`
   supporting Phantom and Backpack. Standard library, no custom implementation.

**Visual sketch:** Wireframes generated during this session (local paths,
session-only — commit to repo or export to Figma for persistence):
- Market grid: 6-player card grid, price + sparkline + stats index per card,
  "Undervalued" filter as key discovery mechanism. Note: wireframe shows real
  player names for readability — devnet uses abstract IDs (Player_23, Player_77)
  pending legal review. Mainnet uses player names once legal structure confirmed.
- Trade screen: bonding curve chart (current position on curve), stats reference
  panel with spread signal, buy/sell widget. Graduation progress bar shown in
  wireframe but hidden in devnet v1 (graduation deferred to v1.5).

Key UI decisions:
- Stats Index shown on every player card — creates the "is this player
  over or underpriced?" question that drives trading behavior
- "Undervalued" filter surfaces players where market price < stats index
- Bonding curve chart shows current position and graduation progress
- Spread signal ("market is X% above/below stats index") shown prominently
  on trade page — this is the core trading signal
- Rough wireframe aesthetic intentional for now; final design to be polished
  post-validation (recommend /design-consultation after devnet demo)

## Open Questions

1. **Oracle source for devnet:** BallDontLie API (free, no license) is the
   provisional choice. Confirm rate limits support the update cadence
   (post-game, ~3-4 updates/day during season).

2. **Graduation threshold:** 69 SOL is the provisional value (mirrors pump.fun).
   Evaluate based on target token supply and expected beta user trading volume.
   May be set lower (e.g., 15-20 SOL) for devnet to make graduation reachable
   in testing.

3. **Graduation phasing:** DECIDED — v1 devnet ships without Raydium graduation.
   Tokens stay on bonding curve indefinitely in devnet. Graduation (bonding
   curve → Raydium CPMM pool) is added in v1.5 before mainnet launch. Raydium
   pool creation via SDK needs a feasibility spike to confirm it can be called
   client-side rather than via Anchor CPI (latter may not be supported).

4. **Solana-compatible sports oracle for mainnet:** Switchboard and Pyth
   primarily support financial price feeds. A custom Switchboard job or
   Chainlink sports adapter needs to be evaluated before mainnet. This is
   the highest-risk dependency — treat it as a long-lead item alongside fundraise.

5. **Legal entity structure:** Before any public launch or token incentives,
   legal review needed on: (a) token issuance structure, (b) right of
   publicity for player names/likenesses on mainnet, (c) NBA data licensing
   for commercial use. Abstract identifiers (Player_23) used on devnet to
   avoid right of publicity exposure during testing.

## Success Criteria

**Devnet validation (2-4 weeks) — bonding curve only, no graduation:**
- [ ] 10 beta users complete at least one buy/sell transaction on devnet
- [ ] At least 3 users describe the stats index as "useful for trading decisions"
      without being prompted
- [ ] At least 1 user says "I would use this if it was on mainnet with real SOL"
- [ ] Bonding curve price moves correctly with each trade (buy → price up,
      sell → price down), with no integer overflow or precision errors observed

**Mainnet v1 (post-validation):**
- [ ] 5 player tokens live on mainnet
- [ ] At least 1 token reaches graduation threshold and migrates to Raydium
- [ ] Total volume > 500 SOL in first 30 days
- [ ] 50+ unique wallets trade at least once

## Engineering Decisions (locked by /plan-eng-review, run 2, 2026-03-29)

Decisions made during the second eng review pass. These refine the architecture
above into implementation-ready specs.

### 1. Scaffolding

Use `npx create-solana-dapp@latest` CLI to generate the Anchor + Next.js monorepo.
Initialize git FIRST (before toolchain install). `.gitignore` + initial commit on Day 0.

### 2. Bonding Curve Reference Implementations

Use pump.fun open-source Anchor clones as reference (NOT copy-paste):
- `m4rcu5o/Solana-pumpfun-smart-contract`
- `g0drlc/Solana-Pumpfun-Smart-Contracts`
- `hexnome/Pumpfun-solana-smart-contract`

Audit reference code for known vulnerabilities: reentrancy via CPI callbacks,
missing signer checks, incorrect PDA bump seed caching, unchecked arithmetic.
Build time reduced from 2-4 weeks to 3-5 days.

### 3. PDA Seeds (locked)

```
BondingCurveAccount: seeds = ["bonding-curve", mint.key().as_ref()], bump
StatsOracleAccount:  seeds = ["stats-oracle", mint.key().as_ref()], bump
```

### 4. Instructions (updated)

```
initialize_curve(player_id, base_price, slope, total_supply)
buy(token_amount, max_sol_in)           # token-denominated buy
buy_with_sol(sol_amount, min_tokens_out) # SOL-denominated buy (NEW)
sell(token_amount, min_sol_out)
update_oracle(index_price_lamports)
collect_fees()                          # admin only, disabled on devnet
```

`buy_with_sol` added for UX: users think in SOL spent, not tokens received.
Both buy instructions must use identical curve math. Frontend picks the right one.
`buy_with_sol` MUST reject when calculated tokens = 0 (dust protection).

### 5. BallDontLie API Strategy

Free tier: 5 requests/minute. Use batch `/stats` endpoint to fetch all player
stats in 1-2 API calls instead of per-player calls. Sufficient for devnet
(~3-4 games/day, 5-10 players).

### 6. WebSocket + Polling Fallback

Frontend uses `accountSubscribe` (WebSocket) for real-time price updates.
On disconnect: auto-reconnect with exponential backoff. Fallback: React Query
polls `getAccountInfo` every 5 seconds until WebSocket reconnects. User never
sees stale data.

### 7. Batch RPC for Market Screen

Use `getMultipleAccounts` to fetch all BondingCurveAccount + StatsOracleAccount
PDAs in one RPC call. Market screen loads in ~200ms instead of ~3s with individual
calls.

### 8. RPC Provider

Helius free tier (100k requests/day) for devnet. Sufficient for 10 beta users.
Tracked in expenses.md at $0/mo. Upgrade to Helius Pro (~$49/mo) post-mainnet.

### 9. Cross-Language Price Parity

The linear curve formula (price = base_price + slope × tokens_sold) exists in
both Rust (on-chain, authoritative) and TypeScript (frontend quote preview).
Automated parity test: feed identical inputs to both, assert identical outputs
within 1 lamport rounding.

### 10. Reserve Invariant (CRITICAL)

After any sequence of buys and sells, the treasury MUST hold enough SOL to
cover all outstanding sell orders at current curve prices. Test with:
- Random 100-trade sequences by multiple users
- Full buy then full sell (treasury → ~0)
- Fuzz testing with random amounts

This is the "does the bank have enough money" test. Catches insolvency bugs.

### 11. Program Upgrade Authority

Devnet: deploy as UPGRADEABLE. Upgrade authority = devnet admin keypair.
Allows code changes without losing state (player tokens, trades survive).
Mainnet: switch to immutable after audit, or use Squads multisig upgrade authority.

### 12. Oracle Wallet Funding

Oracle cron wallet needs SOL to pay tx fees on devnet. Airdrop 2 SOL on setup.
Cost: ~0.000005 SOL per update. Monitor: if oracle stops updating, check wallet balance.
